# 平台架構

## 架構目標

Lemon Platform 的架構目標是建立一個可逐步整合的營運平台，而不是一次性重寫所有既有系統。平台應能吸收既有工具的能力，同時讓未來模組有一致的入口、資料流與 UI 規範。

架構設計應優先考慮穩定性、可追溯性與低切換成本。

## 分層概念

平台可分為四個主要層次：

- 入口層：提供導覽、權限判斷、中心切換與使用者操作起點。
- 模組層：承載 Service、Operations、Finance、Tools、Automation 等中心功能。
- 整合層：串接既有系統、外部資料來源、Google Sheet、API 或人工匯入資料。
- 文件與治理層：保存架構決策、任務模板、AI 規則與維護流程。

分層的目的不是增加複雜度，而是避免中心與工具之間互相混雜。

## 中心架構

每個中心應有自己的首頁、主要任務、資料來源說明與輸出規則。中心內可以有多個功能頁，但應共用平台導覽、狀態顯示與錯誤處理模式。

中心之間可以共享資料，但共享前需定義資料擁有者。例如財務中心可以使用訂單狀態，但不應直接改寫訂單核心資料。

## 整合方式

短期整合可透過連結、文件、匯入匯出與人工檢查完成。中期應整理穩定資料結構與共用流程。長期再視需求建立正式 API 或服務層。

每次整合都應記錄：

- 資料來源。
- 更新頻率。
- 責任系統。
- 錯誤處理方式。
- 是否可由平台直接修改。

## 擴充原則

新增功能前應先確認它屬於哪個中心，是否已有既有系統支援，以及是否需要進入平台主流程。

若功能只是一次性工具，應先放入 Tools Center 或 references，不應直接升級為核心模組。

## 風險控管

平台整合可能帶來資料不一致、權限不清、重複維護與使用者混淆。為降低風險，任何新中心或新流程都應先文件化，再進入實作規劃。
